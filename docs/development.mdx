## Getting Started

Follow these steps to set up context-window for local development.

<Steps>
  <Step title="Clone the repository">
    Clone the context-window repository from GitHub:

    ```bash
    git clone https://github.com/hamittokay/context-window.git
    cd context-window
    ```
  </Step>

  <Step title="Install dependencies">
    Install all required packages using pnpm (recommended) or npm:

    ```bash
    pnpm install
    # or
    npm install
    ```
  </Step>

  <Step title="Set up environment variables">
    Create a `.env` file in the project root for testing:

    ```bash
    # OpenAI Configuration
    OPENAI_API_KEY=sk-...

    # Pinecone Configuration
    PINECONE_API_KEY=...
    PINECONE_INDEX=context-window-test
    PINECONE_ENVIRONMENT=us-east-1
    ```

    <Warning>
      These credentials will be used for running tests. Make sure to use test accounts or create a separate test index.
    </Warning>
  </Step>

  <Step title="Build the project">
    Compile TypeScript to JavaScript:

    ```bash
    pnpm run build
    # or
    npm run build
    ```

    This generates the `dist/` folder with compiled output.
  </Step>
</Steps>

## Development Workflow

### Available Scripts

The project includes several npm scripts for development:

```bash
# Build the project
pnpm run build

# Watch mode for development (auto-rebuild on changes)
pnpm run dev

# Run tests
pnpm test

# Type checking without emitting files
pnpm run check

# Lint code
pnpm run lint

# Format code with Prettier
pnpm run format
```

### Running Tests

<Tabs>
  <Tab title="All Tests">
    Run all unit and e2e tests:

    ```bash
    pnpm test
    ```
  </Tab>

  <Tab title="Specific Test">
    Run a specific test file:

    ```bash
    pnpm test tests/unit/chunker.test.ts
    ```
  </Tab>

  <Tab title="With Coverage">
    Generate test coverage report:

    ```bash
    pnpm test -- --coverage
    ```
  </Tab>
</Tabs>

<Note>
  **Important**: E2E tests require valid API keys in `.env` and will make real API calls to OpenAI and Pinecone.
</Note>

### Testing Your Changes

To test your changes locally:

1. Build the project: `pnpm run build`
2. Link it globally: `npm link`
3. In another project: `npm link context-window`
4. Make changes and rebuild to test

## Project Structure

Understanding the codebase organization:

```
context-window/
├── src/
│   ├── adapters/       # Provider adapters
│   │   ├── ai/         # AI provider interfaces (OpenAI)
│   │   └── store/      # Vector store interfaces (Pinecone)
│   ├── core/           # Core ContextWindow class
│   │   ├── ContextWindow.ts
│   │   └── registry.ts
│   ├── io/             # File ingestion and reading
│   │   ├── ingest.ts
│   │   └── readers.ts
│   ├── util/           # Utilities
│   │   ├── chunker.ts
│   │   └── hash.ts
│   └── types.ts        # TypeScript type definitions
├── tests/
│   ├── unit/           # Unit tests
│   ├── e2e/            # End-to-end tests
│   └── __mocks__/      # Test mocks
├── examples/           # Usage examples
├── docs/               # Documentation
└── dist/               # Compiled output (generated)
```

## Contributing

We welcome contributions! Here's how to contribute:

<Steps>
  <Step title="Create a feature branch">
    ```bash
    git checkout -b feature/your-feature-name
    ```
  </Step>

  <Step title="Make your changes">
    - Write clean, readable code
    - Follow existing code style
    - Add tests for new functionality
    - Update documentation as needed
  </Step>

  <Step title="Run quality checks">
    Before committing, ensure everything passes:

    ```bash
    pnpm run lint    # Check code style
    pnpm run check   # Type check
    pnpm test        # Run tests
    ```
  </Step>

  <Step title="Commit your changes">
    Write clear, descriptive commit messages:

    ```bash
    git add .
    git commit -m "feat: add support for .docx files"
    ```
  </Step>

  <Step title="Push and create a PR">
    ```bash
    git push origin feature/your-feature-name
    ```

    Then open a Pull Request on GitHub with:
    - Clear description of changes
    - Link to related issues (if any)
    - Screenshots/examples (if applicable)
  </Step>
</Steps>

## Code Style Guidelines

### TypeScript

- Use TypeScript strict mode
- Prefer explicit types over `any`
- Use interfaces for public APIs
- Document complex functions with JSDoc comments

### Naming Conventions

- **Files**: camelCase for utilities, PascalCase for classes
- **Functions**: camelCase with descriptive names
- **Types/Interfaces**: PascalCase
- **Constants**: UPPER_SNAKE_CASE

### Testing

- Write tests for all new features
- Aim for high code coverage
- Use descriptive test names
- Mock external API calls in unit tests

## Adding New Features

### Adding a New File Type

To support a new file format (e.g., `.docx`):

1. Add reader function to `src/io/readers.ts`
2. Install necessary parsing library
3. Add file extension to supported list
4. Add tests to `tests/unit/readers.test.ts`
5. Update documentation

### Adding a New AI Provider

To add support for a new AI provider:

1. Create interface in `src/adapters/ai/types.ts`
2. Implement adapter in `src/adapters/ai/[provider].ts`
3. Update type definitions
4. Add tests
5. Update documentation

### Adding a New Vector Store

To support a new vector store:

1. Define interface in `src/adapters/store/types.ts`
2. Implement adapter in `src/adapters/store/[store].ts`
3. Handle initialization and CRUD operations
4. Add comprehensive tests
5. Update documentation

## Debugging

### Enable Verbose Logging

Add console logs or use a debugger:

```typescript

console.log("Debug info:", variableName);
```

### VS Code Debugging

Create `.vscode/launch.json`:

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "node",
      "request": "launch",
      "name": "Debug Current Test",
      "program": "${workspaceFolder}/node_modules/vitest/vitest.mjs",
      "args": ["${file}"],
      "console": "integratedTerminal"
    }
  ]
}
```

## Release Process

For maintainers releasing new versions:

1. Update version in `package.json`
2. Update CHANGELOG.md with release notes
3. Run `pnpm run build` to generate fresh dist
4. Commit changes: `git commit -m "chore: release v1.x.x"`
5. Create git tag: `git tag v1.x.x`
6. Push with tags: `git push origin main --tags`
7. Publish to npm: `npm publish`
